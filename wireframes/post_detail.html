MLOps 확장하기: Jupyter Notebook에서 프로덕션 클러스터까지

Jaehyeong
DevOps Engineer
calendar_today 2023년 10월 12일
schedule 15분 소요

데이터 과학 프로젝트가 실험실 환경(Jupyter Notebook)을 벗어나 실제 프로덕션 환경으로 이동할 때, 많은 팀이 '재현성'과 '확장성'이라는 벽에 부딪힙니다. 모델을 개발하는 것과 모델을 서빙하고 지속적으로 관리하는 것은 전혀 다른 차원의 문제입니다.

이 글에서는 Kubeflow와 최신 CI/CD 파이프라인을 활용하여, 데이터 과학자가 작성한 노트북 코드가 어떻게 자동으로 프로덕션 파이프라인으로 변환되고 배포될 수 있는지 그 과정을 상세히 다룹니다.

왜 Notebook만으로는 부족한가?

Jupyter Notebook은 탐색적 데이터 분석(EDA)과 초기 모델링에는 탁월한 도구입니다. 하지만 다음과 같은 이유로 프로덕션 환경에는 적합하지 않습니다:

  • 버전 관리의 어려움: 노트북 파일(ipynb)은 JSON 형식이어서 Git과 같은 도구로 변경 사항을 추적하기 어렵습니다.
  • 의존성 격리 부족: 로컬 환경에 설치된 라이브러리 버전이 프로덕션 환경과 다를 경우 동작을 보장할 수 없습니다.
  • 테스트 자동화 불가능: 단위 테스트나 통합 테스트를 체계적으로 수행하기 어렵습니다.
"MLOps의 핵심은 모델 코드뿐만 아니라 데이터, 환경 설정, 파이프라인 구성까지 모두 코드로 관리(Infrastructure as Code)하는 것입니다."

Kubeflow Pipelines로 전환하기

Kubeflow는 Kubernetes 위에서 실행되는 머신러닝 툴킷으로, 복잡한 ML 워크플로우를 오케스트레이션하는 데 최적화되어 있습니다. Python SDK를 사용하면 파이썬 함수를 컴포넌트로 쉽게 변환할 수 있습니다.

pipeline.py Python 
import kfp
from kfp import dsl
@dsl.component
def train_model(data_path: str, epochs: int) -> str:
    # 훈련 로직 구현
    return "model_v1.pt"
@dsl.pipeline(
    name="training-pipeline",
    description="A simple training pipeline"
)
def my_pipeline(data: str, epochs: int = 10):
    train_task = train_model(data_path=data, epochs=epochs)

위 코드에서 볼 수 있듯이, @dsl.component 데코레이터를 사용하여 일반적인 파이썬 함수를 Kubeflow 컴포넌트로 변환합니다. 이 컴포넌트들은 각각 독립된 컨테이너에서 실행되므로 환경 격리 문제가 해결됩니다.

결론

실험 단계에서 프로덕션으로의 전환은 MLOps의 가장 큰 병목 구간 중 하나입니다. Kubeflow와 같은 도구를 도입하고 GitOps 방식의 CI/CD를 구축함으로써, 데이터 과학자는 모델 성능 향상에 집중하고 엔지니어는 안정적인 서빙 인프라를 제공하는 선순환 구조를 만들 수 있습니다.
